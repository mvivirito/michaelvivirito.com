What This Is

Open with a single short paragraph that says what the post covers and what problem it solves. The reader should know inside two sentences whether to keep reading.

Why Bother

One or two paragraphs on motivation. Often this is "the obvious commercial option doesn't fit my constraints" or "the standard guide skips the part I kept getting stuck on". Be specific; vague motivation reads like filler.

Bill of Materials

  • Item one: what it is and why this specific one.
  • Item two: same.
  • Item three: same.

Step One: First Phase

Explain the intent of the step before the commands, not after. The reader is going to run these in a terminal in 30 seconds; they need the why first so they can decide whether your situation matches theirs. 

# Comments explain non-obvious flags
$ command --flag value
$ another-command | piped-thing

Step Two: Second Phase

Same shape. If a step has a single load-bearing line ("this one knob is the whole post"), call it out in its own paragraph after the code block rather than burying it in a list. 

$ short snippet
$ another snippet

Step Three: Configuration File

For full config files, name the path in a comment at the top so a reader skimming the post can identify where the snippet belongs. 

# /etc/example.conf
key = "value"
another_key = "another value"

Smoke Test

Three or four commands the reader can run to confirm the setup works. Show expected output where it adds clarity. 

$ test-command
expected output here
$ another-test
expected output here
